# Industry OS M6 真实用户验证方案

## 目标与边界

验证 3–5 名目标用户能否在没有现场指导的情况下完成“输入行业 → 等待研究 → 阅读报告 → 区分事实与假设 → 分享或下载”的核心流程。

本阶段验证产品可用性，不人工补充行业研究数据，不把用户意见改写成公开市场事实。M6.1 只准备方案；联系、邀请或招募用户必须等 M6.2 获得 L5 授权。

## 测试对象

- 3–5 名需要做电商行业或竞品研究的经营者、产品负责人或研究人员。
- 至少 3 人不是本项目开发者，也没有提前看过操作说明。
- 每人使用一个自己熟悉但不含私人数据的公开行业或品类。
- 不收集店铺账号、客户名单、订单、联系方式、支付信息、Cookie、验证码或付费墙内容。

## 单人任务

1. 打开 `https://research.playgamelab.cn/industry-research`。
2. 只根据页面提示输入行业、品类、市场和项目名称，不接受现场操作指导。
3. 每人最多启动一次 `public_web` 公开市场研究；不使用登录态、人工补料或私人数据。
4. 等待流程结束，判断报告是否明确区分已确认事实、推断、未验证机会和数据缺口。
5. 用自己的话说出一条已确认事实、一条待验证机会，以及下一步应该验证什么。
6. 完成一次分享或下载，并说明报告是否足以支持下一步研究，而不是直接支持商业化继续或停止结论。

## 成功指标

- 至少 3 名用户全程无需现场指导完成输入、等待、阅读和分享/下载。
- 至少 3 名用户能正确区分“公开证据支持的事实”和“尚待验证的机会假设”。
- 至少 3 名用户能找到来源或证据入口，并理解数据缺口不等于停止商业化。
- 核心流程没有 P0/P1 问题：密钥或私人数据暴露、越权访问、无法完成、报告事实无来源、事实与假设混淆、分享/下载失败。
- 每名用户只计一次首次独立尝试；现场提示后的成功不计入 C4。

## 失败与停止条件

- 任意密钥、账号态、私人数据或未授权联系方式被采集或暴露：立即停止全部测试。
- 同一核心步骤连续两人失败，或生产出现重复 5xx、任务失控、重复扣费迹象：停止后先修复。
- 报告再次把证据缺口写成“停止商业化”，或把未验证机会写成已确认结论：按 P1 处理并停止扩招。
- 单人超过一次 live run、出现登录墙/验证码/付费墙绕过请求，或超出既定公开市场预算：不继续执行。

## 记录模板

每名参与者只记录匿名编号、角色类型、测试品类、开始/结束时间、各步骤成功与否、是否需要提示、事实/假设复述结果、P0/P1/P2 问题和一句原话反馈。不记录姓名、联系方式、公司秘密、客户或订单数据。

记录写入 `outputs/industry-data-report-loop/m6.3/`，不进入公开报告证据库、Supabase 行业事实表或 zvec 行业知识索引。

## 权限门

M6.2 需要单独 L5 授权：仅邀请 3–5 名明确的目标用户参加上述测试，并使用用户确认的沟通渠道。该授权不包含公开招募、群发、付费承诺、销售沟通、收集私人业务数据或把反馈当行业事实。
